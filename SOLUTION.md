# Telemetry Ingestion Core

### Description

A small vertical slice of a telemetry ingestion pipeline, using ASP.NET Core (.NET 8), SQLite via EF Core, Typescript React and GitHub Actions.

- **Filename**: SOLUTION.md
- **Author**: Willa Charlotte Lyle
- **Date**: 29 April 2026
- **Stack**: .NET 8, ASP.NET Core, EF Core (SQLite), xUnit, GitHub Actions

## Summary

A simple, tightly scoped telemetry pipeline that:

- Accepts single telemetry readings into a backend service
- Persists them to a lightweight SQL store
- Exposes a simple query interface
- Builds and tests via a single CI pipeline
- (Optional) Builds backend service into a docker image
- (Optional) Displays readings via a frontend service

Example telemetry payload:

```json
{
  "tenantId": "acme",
  "deviceId": "dev-123",
  "type": "water_level",
  "value": 1.23,
  "unit": "m",
  "battery": 62,
  "signal": -85,
  "recordedAt": "2025-01-10T10:15:00Z",
  "externalId": "r-789"
}
```

## Requirements and Constraints

The requirements given for the service are listed as follows.

### Backend

- ASP.NET Core (.NET 8).
- Single-reading ingest only (one reading per POST request).
- Validation for required fields with sensible ranges.
- Duplicate prevention for when externalId is repeated for the same tenant.
- Filtering and pagination for queries.
- Structured logging.
- Simple health endpoint for basic readiness/liveness.
- One or two environment driven configurations.
- Simple domain rules.
- Computation of a status object (e.g. batteryLow when below configured threshold).
- Query endpoint with filtering by deviceId, type, and time range, plus pagination.
- Automated tests focused on core domain logic and at least one happypath API flow.
- Minimal APIs and controllers.
- Evolvable model with clear API shape without overengineering.
- (Optional) Correlation ID propagation if provided by the client.

### DB

- SQLite persistence of readings (using EF core).
- Reasonable schema and mapping.

### Frontend (Optional)

- Readonly table with recent readings (last N or fixed recent).
- Single loading and error states.
- Typed API models.
- Environment driven API base url.
- (Optional) Device and type filters.
- (Optional) Simple submit form to POST a single reading.

### CI/CD

- GitHub Actions.
- Builds and tests project.
- Published test results/artifacts.
- (Optional) Docker image for the backend.
- (Optional) Basic caching for package restores.

### Abbreviated Data Flow

1. Client POSTs a single reading.
2. Service validates and maps entity.
3. Repository attempts insert and states if conflicted on tenantId and externalId.
4. On success, Service calcuates and adds the BatteryLow status object and API returns 201 with the persisted entity.
5. Consumers query via GET /api/telemetry with filters and pagination.
6. Service queries the repository for the results and adds each relevant BatteryLow status object.
7. API returns 200 with queried entities.

### Backend API (ASP.NET Core .NET 8)

This section outlines the models used throughout the service

#### Data Models

The following data models are used for the EF context, reading ingestion and query response, respectively. For the sake of brevity, only the TelemetryReading model will show descriptions for each property. The models TelemetryDto and TelemetryView following inherit the same descriptions.

TelemetryReading (EF entity)

- `Id: Guid (PK)` - Unique identifier generated by the server for every new reading.
- `TenantId: string (indexed)` - Tenant identifier owning the telemetry reading.
- `ExternalId: string? (indexed)` - External identifier provided by the device or upstream system. Unique per TenantId when present.
- `DeviceId: string (indexed)` - Device identifier that produced the reading.
- `Type: string` - Measurement type (for example "temperature", "water_level", etc.).
- `Value: double` - Numeric measurement value for the reading.
- `Unit: string` - Unit of measurement for the reading (for example "C", "m", "%", etc.).
- `Battery: int` - Battery level reported by the device as a percentage.
- `Signal: int` - Signal strength reported by the device. Expects negative dBm values (e.g. -70).
- `RecordedAt: DateTimeOffset` - Timestamp when the measurement was recorded on the device.
- `CreatedAt: DateTimeOffset` - Timestamp when the measurement was created in the database. Generated by the server.

TelemetryDto (Reading ingestion)

- `TenantId: string` - Required. Max length is 200.
- `ExternalId: string?` - Max length is 200.
- `DeviceId: string` - Required. Max length is 200.
- `Type: string` - Required. Max length is 200.
- `Value: double` - Required.
- `Unit: string` - Required. Max length is 50.
- `Battery: int` - Required. Accepted range is between 0 and 100.
- `Signal: int` - Required. Accepted range is between -200 and 0.
- `RecordedAt: DateTimeOffset` - Required.

TelemetryView (View model for responses)

- `TenantId: string`
- `ExternalId: string?`
- `DeviceId: string`
- `Type: string`
- `Value: double`
- `Unit: string`
- `Battery: int`
- `BatteryLow: bool` - Computed status object indicating the device's battery is low.
- `Signal: int`
- `RecordedAt: DateTimeOffset`
- `CreatedAt: DateTimeOffset`

PaginationMetadata (Model for information about pagination)

- `PageNumber: int` - The current page number of the pagination.
- `PageCount: int` - The total number of pages available.
- `PageSize: int` - The page size of the pagination.

TelemetryPaginationView (Model for responses that contain telemetry readings and pagination)
- `TelemetryReadings: IEnumerable<TelemetryView>` - The queried telemetry readings.
- `PaginationMetadata: PaginationMetadata` - Metadata about pagination.

TelemetryReadingsPaginationData (Model for the repository to return telemetry readings and pagination data)
- `TelemetryReadings: IReadOnlyList<TelemetryReading>` - The queried telemetry readings.
- `PaginationMetadata: PaginationMetadata` - Metadata about pagination.

##### Considerations

Due to the tight scope of this project, some assumptions have been made about the types and validations of the models.

###### Specificity of Properties

Ideally, properties such as TenantId and Unit should be restricted to a range of acceptable values for each specific use case. We would only want to accept ingestion from vetted tenants, or readings containing valid units of measurement. Such validations could be performed using configurable lists of acceptable values or with enums in code, but have been omitted because of the small scope and abstract nature of this project.

#### API Layout

##### Telemetry Endpoints (Minimal API controllers used)

- POST /api/telemetry
  - Accepts single TelemetryDto object.
  - Returns 400 BadRequest if DTO is not present.
  - Validates required fields and ranges and returns 400 BadRequest with validation errors in body when invalid.
  - Prevents duplicate insert if ExternalId exists for same TenantId and returns 409 Conflict for such error.
  - Computes BatteryLow status using configured threshold.
  - Returns 201 Created with TelemetryView object in body.
  - Catches all other exceptions and returns 500 with details in body

- GET /api/telemetry
  - Optional query parameters:
    - tenantId
    - deviceId
    - type
    - from (ISO DateTime)
    - to (ISO DateTime)
    - page (default 1, configurable)
    - pageSize (default 50, max 500, both configurable)
  - Returns 400 BadRequest if any query parameter is invalid (determined by the same validation rules as DTO).
  - Returns 200 Ok with paged list of TelemetryView objects and pagination metadata.
  - Sorted by RecordedAt, with most recent recordings first
  - Catches all other exceptions and returns 500 with details in the body

##### Health Endpoints (Minimal API controllers used)

- GET /health/live
  - Liveness probe: returns 200 Ok.

- GET /health/ready
  - Readiness probe: checks DB is accessible (with simple SELECT 1) and returns 200 Ok or 503 ServiceUnavailable.

##### Considerations

Due to small scope of the project, a fairly minimal API is provided. Future changes could be made to:

- Provide multiple-reading ingestion.
- Query for a specific reading by Id.
- Query for all readings (may be resource intensive).
- Delete or alter readings.

#### Services

The application uses the following internal services:

- TelemetryService
  - Responsible for processing telemetry transactions between the API layer and database layer.
  - Validates data transfer objects, applies domain rules and transforms between view, database and DTO models.

- HealthService
  - A simple health service that checks application readiness with a simple CheckDb method.
  - Not really needed for a project with this scope, but is provided for further extensibility.

#### Validation and Business Rules

- Validation on ingest:
  - Fields are validated against requirements specified in the TelemetryDto above and errors are returned if invalid.
  - Domain-specific range checks could be added as the scope requires them.

- Duplicate prevention:
  - If ExternalId provided:
    - Try insert with unique constraint on (TenantId, ExternalId).
    - If violation, return 409 Conflict and id of existing reading (if known).
    - Due to limitations in SQLite and the requirement that externalId can be null or empty, uniqueness will be enforced in code transactions.
  - If ExternalId not provided:
    - No dedupe, each POST produces a new record.

- BatteryLow computation:
  - Returns true if ingested battery value is lower than the configured threshold.
  - Threshold determined by app option BatteryLowThreshold (int, default 20).

- Logging:
  - Structured JSON logs with JSON serialization
  - Include fields: tenantId, deviceId, externalId (if present), and any other fields related to the log.
  - Log levels: Information for successful ingests and queries, Warning for validation problems and Error for exceptions.
  - Logs are routed to Console for development, and in production can route to a logging collector.

#### Tests (xUnit)

Due to the small scope of the project, a fairly minimal testing coverage has been chosen. Strategy will focus on a handle of unit tests for core functionality and integration tests for happy flows and a couple negative flows.

##### Test Strategy

- Unit tests:
  - Positive and negative DTO validation.
  - Common repository transactions:
    - Positive transactions with filters and pagination applied.
    - Positive transactions with autogenerated fields sufficiently populated.
    - Negative transations such as ArgumentNullException when reading is null and domain exception DuplicateExternalIdException when two readings of the same tenantId and externalId are added.

- Integration tests:
  - WebApplicationFactory-based tests using in-memory SQLite database.
  - Happy flows for the simple health checks.
  - Happy flow for posting a reading and retrieving it from persistence.
  - Negative flow for when a DTO fails validation.
  - Negative flow for when a duplicate externalId is found for the same tenantId.

#### Configuration

The following options are configurable for the service:

App Options:

- int BatteryLowThreshold - The threshold used for calculating the batteryLow status object. (default = 20)
- int MaxPageSize - The maximum page size allowed in pagination from the query endpoint. (default = 500)
- int DefaultPageSize - The default page size for pagination from the query endpoint. (default = 50)

Connection Strings:

- string TelemetryDb - The connection string used for interacting with the SQLite database (default = "Data Source=./wwwroot/content/data/TelemetryDatabase.db")

Logging:

- Configuration of the log levels for the application to display.

### Persistence (SQLite + EF Core)

DbContext:

- TelemetryContext : DbContext with DbSet<TelemetryReading> for readings.
- Properties contain the same restraints as in the DTO.
- For simplicity, the GUID Id and DateTimeOffset CreatedAt will be generated by the server and not the database. This is ideal for single instance deployments.

Migrations & initialization:

- Uses EF Core migrations for schema evolution.
- For development, `dotnet-ef` tool is used for creating the database and applying migrations.
- For tests, SQLite with the connection string "DataSource=:memory:" is used to avoid transacting with any production database.

Transactions:

- Uses a repository design pattern to handle DbContext transactions with the database. Contains the following:
  - An insert method that creates a telemetry reading in the database.
    - Checks for null readings.
    - Checks for duplicate externalId's for each tenantId.
    - Leaves further object validation to the telemetry service that interacts with the repository.
  - A query method that returns a paginated list of telemetry readings.
    - Uses the following optional filters:
      - string tenantId
      - string deviceId
      - string type
      - DateTimeOffset from
      - DateTimeOffset to
    - Uses the following pagination properties:
      - int page
      - int pageSize
    - Orders the list by most recently recorded.

Indexes:

- Has indexes TenantId, DeviceId and RecordedAt to optimize common queries.
- SQLite doesn't support filtered unique indexes via EF migrations directly.
  - Add a regular index { TenantId, ExternalId } and prefer adding the filtered-unique index in a migration with SQL for production.

#### Considerations

SQLite has many limitations, especially for multi-instance or scaled up deployments. Would recommend moving to PostgreSQL or SQL Server when scaling for better multi-node support. For the small scope of this project it is chosen for its simplicity and low operation cost.

Since SQLite is file-based, production deployments should:
  - Place the .db file on a persistent disk.
  - Ensure regular backups of the file.
  - Use WAL (Write-Ahead Logging) mode for better concurrency, performance, and durability.

### CI Pipeline (GitHub Actions)

For every pull request or push to main the CI/CD pipeline will run a job that:

- Builds the project and shows any errors encountered.
- Runs the project tests and outputs them to a TRX file.
- Interprets the TRX file and publishes the results into a test report.
